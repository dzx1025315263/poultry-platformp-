# 全球外贸数据协作平台 — 多行业扩展架构方案

## 一、现状诊断：行业绑定点全景

经过对平台全部代码的逐文件审查，当前与"禽业"硬编码绑定的部分可以归纳为以下 6 个层面：

| 层面 | 绑定点数量 | 涉及文件 | 典型示例 |
|------|-----------|---------|---------|
| **UI 文案与标题** | ~30 处 | DashboardLayout, Home, TradePage, WeeklyReportPage, FeishuSettingsPage 等 | "全球禽业数据平台""全球禽肉进口商""全球肉鸡行业外贸深度分析" |
| **企业分类标签** | ~3 处 | SearchPage, ExportPage | `ROLE_TAGS = ["全产业链巨头", "进口商/贸易商", "加工商", "冷链物流", "养殖/饲料", ...]` |
| **市场洞察内容** | ~15 处 | InsightsPage | 各大洲的禽肉市场分析文案（沙特禁令、清真认证、冻鸡爪需求等） |
| **贸易数据模型** | ~10 处 | schema.ts, db.ts, routers.ts, TradePage | 表名 `poultry_trade_data`、HS Code 默认值 `0207`、函数名 `getPoultryTradeData` |
| **AI Prompt 模板** | ~6 处 | routers.ts, weeklyReportCron.ts | "你是一位资深的全球禽肉行业外贸分析师""禽肉行业外贸客户匹配专家" |
| **项目元数据** | ~3 处 | package.json, index.html, VITE_APP_TITLE | 项目名 `poultry-platform`、页面标题"全球禽业数据协作平台" |

**核心发现：** 平台的底层架构（用户系统、CRM、团队协作、邮件自动化、A/B 测试、待办事项、数据导出、备份等）是完全行业无关的。行业绑定集中在"展示层文案"和"数据模型命名"两个维度，改造成本可控。

---

## 二、多行业扩展的两种路径

根据业务规划的不同阶段，有两种可选路径：

### 路径 A：配置化单实例（推荐起步方案）

> 一套代码部署多个实例，每个实例通过配置文件切换行业。适合初期快速验证 2-3 个行业。

**核心思路：** 将所有行业特定内容抽取到一个 `industry-config.ts` 配置文件中，通过环境变量 `VITE_INDUSTRY` 选择加载哪个行业配置。

```
shared/
  industry-config/
    poultry.ts      ← 禽业配置
    seafood.ts      ← 水产配置
    beef.ts         ← 牛肉配置
    index.ts        ← 根据 VITE_INDUSTRY 导出当前配置
```

每个配置文件定义统一的接口：

```typescript
export interface IndustryConfig {
  // 基础信息
  name: string;              // "全球禽业数据协作平台"
  shortName: string;         // "禽业数据平台"
  industry: string;          // "poultry"
  icon: string;              // 行业图标

  // 企业分类
  roleTags: string[];        // ["全产业链巨头", "进口商/贸易商", ...]
  
  // 贸易数据
  hsCode: string;            // "0207"
  hsCodeDesc: string;        // "禽肉及可食用杂碎"
  tradeDataLabel: string;    // "禽肉进口数据"
  
  // AI Prompt
  analystRole: string;       // "全球禽肉行业外贸分析师"
  reportTitle: string;       // "全球肉鸡行业外贸深度分析报告"
  matchExpertRole: string;   // "禽肉行业外贸客户匹配专家"
  
  // 市场洞察
  regionInsights: Record<string, InsightItem[]>;
  
  // 飞书通知
  feishuBotName: string;     // "禽业数据助手"
  feishuGroupName: string;   // "禽业数据通知"
}
```

**改造工作量估算：**

| 改造项 | 工作量 | 说明 |
|--------|--------|------|
| 创建 `IndustryConfig` 接口和禽业默认配置 | 2 小时 | 将现有硬编码提取为配置 |
| 替换前端 ~30 处硬编码文案 | 3 小时 | 用 `config.name` 替换 "全球禽业数据平台" 等 |
| 替换后端 ~6 处 AI Prompt | 1 小时 | 用 `config.analystRole` 替换角色描述 |
| 贸易数据表泛化（重命名 + HS Code 参数化） | 2 小时 | `poultry_trade_data` → `trade_data`，HS Code 从配置读取 |
| 编写第二个行业配置（如水产） | 1 小时 | 填写 `seafood.ts` 的各字段 |
| 测试验证 | 2 小时 | 切换配置后全流程验证 |
| **合计** | **~11 小时** | |

### 路径 B：多租户 SaaS 架构（长期规划）

> 一套代码、一个数据库实例，通过租户隔离支持多个行业同时运行。适合平台化运营 5+ 个行业。

**核心思路：** 在数据库层面增加 `industry_id` 字段，所有数据查询自动附加行业过滤条件。

```
新增表：
  industries (id, name, config_json, created_at)
  
现有表增加字段：
  companies.industry_id → 关联行业
  favorites.industry_id → 按行业隔离收藏
  trade_data.industry_id → 按行业存储贸易数据
  weekly_market_reports.industry_id → 按行业生成报告
```

**路径 B 相比路径 A 的额外工作：**

| 额外改造项 | 工作量 | 说明 |
|-----------|--------|------|
| 数据库 schema 增加 `industry_id` 字段 | 3 小时 | 涉及 ~15 张表的迁移 |
| 后端查询层增加行业过滤中间件 | 4 小时 | tRPC context 注入当前行业 |
| 前端增加行业切换器 | 3 小时 | 顶部导航或登录时选择行业 |
| 数据导入脚本支持多行业 | 2 小时 | 导入时指定 industry_id |
| 行业管理后台 | 4 小时 | 管理员创建/编辑行业配置 |
| **额外合计** | **~16 小时** | 在路径 A 基础上追加 |

## 三、推荐实施策略

建议采用 **渐进式演进** 的方式：

**第一阶段（立即可做）：路径 A — 配置化抽取**

这一阶段的目标是让现有代码具备"换皮"能力。具体步骤如下：首先创建 `shared/industry-config/` 目录和 `IndustryConfig` 类型接口，然后将禽业的所有硬编码文案、标签、Prompt 提取到 `poultry.ts` 配置文件中，接着在前端和后端代码中用 `import { config } from '@shared/industry-config'` 替换所有硬编码引用，最后将贸易数据表从 `poultry_trade_data` 重命名为 `trade_data`，HS Code 改为从配置读取。完成后，只需复制 `poultry.ts` 为 `seafood.ts`，修改字段值，设置 `VITE_INDUSTRY=seafood`，即可部署一个水产行业的独立实例。

**第二阶段（验证 2-3 个行业后）：路径 B — 多租户升级**

当确认多个行业的商业模式成立后，再投入精力做数据库层面的租户隔离，支持单实例多行业并行运行。

**第三阶段（规模化后）：平台化**

增加行业模板市场、自助开通、独立域名绑定等能力，让新行业的接入成本趋近于零。

## 四、需要预留的通用化设计清单

无论选择哪条路径，以下设计原则应立即纳入开发规范：

| 设计原则 | 具体做法 | 当前状态 |
|---------|---------|---------|
| **文案不硬编码** | 所有面向用户的行业文案从配置文件读取 | 需改造（~30 处） |
| **分类标签可配置** | 企业角色标签 `ROLE_TAGS` 从配置读取 | 需改造（3 处） |
| **HS Code 参数化** | 贸易数据的 HS Code 不写死 `0207` | 需改造（2 处） |
| **AI Prompt 模板化** | 所有 LLM 调用的 system prompt 从配置读取 | 需改造（6 处） |
| **数据表命名泛化** | 避免表名包含行业词（如 `poultry_trade_data` → `trade_data`） | 需改造（1 张表） |
| **企业字段通用化** | `purchasePreference`、`mainProducts` 等字段名已经是通用的 | 已满足 |
| **CRM 流程通用** | 收藏夹、生命周期、邮件、A/B 测试等流程不含行业逻辑 | 已满足 |
| **团队协作通用** | 团队管理、权限、活动流等不含行业逻辑 | 已满足 |

## 六、总结

当前平台的核心架构（用户系统、CRM、团队协作、邮件自动化、AI 推荐、数据导出、备份等 **占代码量 85% 以上**）已经是行业无关的。行业绑定集中在展示层文案（~30 处）、AI Prompt（~6 处）和一张贸易数据表的命名上，改造成本约 11 小时。建议优先执行路径 A（配置化抽取），快速验证第二个行业，再根据业务发展决定是否升级到多租户架构。
